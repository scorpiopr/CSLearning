常量折叠（Constant Folding）是 C++ 编译器在编译时，编译器如果发现某个表达式的值是恒定不变的，就会直接计算出该表达式的结果，并用这个结果替换掉原来的表达式。 这样在程序运行时，CPU 就不需要再重复做这些无意义的数学运算，从而提升运行效率。
------------------------------
## 1. 基础的常量折叠（数学表达式）
这是最简单的场景，任何现代 C++ 编译器都会默认执行：

// 优化前：你写的代码
```cpp
int seconds = 60 * 60 * 24;
```
// 优化后：编译器实际生成的机器码效果.
```cpp
int seconds = 86400; 
```
在运行期，程序直接加载 86400，没有任何乘法开销。
------------------------------
## 2. C++ 中结合 const 的常量折叠（高频考点）
在 C++ 中，当一个 const 变量用字面量初始化时，编译器会把它加入到符号表（Symbol Table）中。后续代码中所有用到这个变量的地方，都会被编译器直接替换为它的字面量值。
这就导致了一个面试中非常著名的“诡异现象”：
## 🚨 经典面试代码陷阱
```cpp
#include <iostream>
int main() {
    const int a = 10;          // 触发常量折叠，编译器记住 a 就是 10
    int* p = const_cast<int*>(&a); // 强行转为普通指针，骗过编译器
    
    *p = 20;                   // 试图通过指针修改内存中的值
    
    std::cout << "a = " << a << std::endl;   // 输出 a 的值
    std::cout << "*p = " << *p << std::endl; // 输出 p 指向的值
    std::cout << "地址: " << &a << " vs " << p << std::endl; // 输出地址
}
```
## 运行结果分析
如果你在现代 C++ 编译器下运行这段代码，你会看到一个反直觉的结果：
```cpp
a = 10
*p = 20
```
地址: 0x7ffee3b3a7bc vs 0x7ffee3b3a7bc

## 为什么地址一样，值却不同？

   1. *p = 20 确实成功了：因为通过地址强行修改，这块内存里的数值确实从 10 变成了 20。所以打印 *p 时拿到的是内存里最新的 20。
   2. a 打印出来依然是 10：因为 “常量折叠” 优化。编译器在编译阶段，看到 std::cout << a 时，就把 a 直接替换成了数字 10。这行代码在编译后实际变成了 std::cout << 10。编译器根本没有去内存里读取 a 的值。

## 3. 常量折叠与宏定义（#define）的区别
既然常量折叠也是“直接替换”，那它和 C 语言的 #define 有什么区别？

* #define（宏替换）：发生在预处理阶段。它只是单纯的文本复制粘贴，没有类型检查，极易引发拼写错误或作用域污染。
* 常量折叠：发生在编译优化阶段。它不仅进行了替换，而且变量是有明确类型的，会严格执行 C++ 的类型安全检查，同时遵循作用域规则（如类内、命名空间内）。
