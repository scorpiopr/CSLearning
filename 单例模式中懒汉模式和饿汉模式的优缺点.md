## 一、 饿汉模式 (Eager Initialization)
核心特点：迫不及待。在类加载或程序启动（进入 main 函数之前）时，就已经把静态实例创建好了。
## 💻 C++ 经典实现
```cpp
class EagerSingleton {private:
    EagerSingleton() = default; // 私有构造
    static EagerSingleton instance; // 静态成员声明
public:
    EagerSingleton(const EagerSingleton&) = delete;
    EagerSingleton& operator=(const EagerSingleton&) = delete;

    static EagerSingleton& getInstance() {
        return instance;
    }
};
// 在全局作用域初始化，main 函数执行前就会构造完成EagerSingleton EagerSingleton::instance;
```
## 优点

   1. 天然线程安全：由于实例在 main 函数启动、多线程环境尚未建立之前就已经初始化完毕，因此绝对不会存在多线程抢夺创建实例导致的数据竞争问题。
   2. 执行效率高：获取实例的 getInstance() 极其简单，只是返回一个现成的引用，没有任何锁的开销。

## 缺点

   1. 可能导致资源浪费：无论该实例后续是否会被用到，它都会在启动时占用内存。如果该类的初始化需要耗费大量资源（如连接数据库、加载大型配置文件），会拉长程序启动时间。
   2. 静态初始化顺序问题（Static Initialization Order Fiasco）：如果两个不同的饿汉单例类存在相互依赖关系（例如单例 A 的构造需要用到单例 B），由于 C++ 无法保证不同编译单元（不同 .cpp 文件）间全局静态变量的初始化顺序，极易引发未定义行为（崩溃）。

## 二、 懒汉模式 (Lazy Initialization)
核心特点：不推不爽。实例直到第一次被调用 getInstance() 时，才会真正触发构造。
## 💻 C++11 及更新版本的最佳实现（Meyers' Singleton）
在现代 C++ 中，最优雅、最安全的懒汉模式是利用 C++11 局部静态变量的特性，无需手动加双重检查锁（DCL）。
```cpp
class LazySingleton {private:
    LazySingleton() = default; // 私有构造
public:
    LazySingleton(const LazySingleton&) = delete;
    LazySingleton& operator=(const LazySingleton&) = delete;

    static LazySingleton& getInstance() {
        // C++11 标准保证：当多个线程同时试图初始化同一个局部静态变量时，
        // 编译器底层会自动加锁保护，确保初始化过程是线程安全的。
        static LazySingleton instance; 
        return instance;
    }
};
```
## 优点

   1. 延迟加载（Lazy Loading）：只有在真正需要用到时才创建对象，不占用多余的启动内存，完美节约系统资源。
   2. 规避了初始化顺序冲突：由于实例是在运行时通过函数调用动态触发构造的，因此不会发生饿汉模式中全局静态变量相互依赖导致的启动崩溃问题。

## 缺点（针对老版本 C++ 而言）

   1. 线程安全实现较复杂（C++11 之前）：在旧版 C++ 中，为了保证多线程下只创建一个实例，必须手动编写复杂的双重检查锁定（Double-Checked Locking, DCL），并且需要引入内存屏障（std::atomic），极易写出含有细微 Bug 的代码。
   2. 微小的运行期开销：每次调用 getInstance() 时，底层编译器都需要检查一个隐式的布尔标志位，以判断局部静态变量是否已经被初始化（虽然现代 CPU 进行了极大的分支预测优化，开销微乎其微）。

## 📊 核心对比汇总表

| 维度 | 饿汉模式 (Eager) | 懒汉模式 (Lazy) |
|---|---|---|
| 创建时机 | 程序启动时（main 之前） | 第一次使用时（运行时延迟加载） |
| 线程安全 | 天然安全，无需额外机制 | C++11 靠局部静态变量保证安全；旧版本需加锁 |
| 启动速度 | 较慢（如果有很多大对象单例） | 极快 |
| 内存使用 | 无论用不用，始终占用内存 | 用时才分配，不用不占内存 |
| 工程推荐度 | 很少使用（有安全隐患） | 强烈推荐（使用 Meyers 局部静态变量写法） |

## 🛠️ 面试官必问：老版本的懒汉模式如何实现多线程安全？
如果面试官追问：“在不支持 C++11 的老环境中，或者转成 Java/C# 语言，你怎么写安全的懒汉模式？”
你应当立刻提出 双重检查锁定 (Double-Checked Locking) 的方案。以 C++11 模拟旧环境下的原子操作锁（防止指令重排）写法：
```cpp
#include <mutex>#include <atomic>
class DclSingleton {private:
    static std::atomic<DclSingleton*> m_instance;
    static std::mutex m_mutex;
    DclSingleton() = default;
public:
    static DclSingleton* getInstance() {
        DclSingleton* tmp = m_instance.load(std::memory_order_acquire);
        if (tmp == nullptr) { // 第一次检查：若已创建则直接跳过锁，保证高效
            std::lock_guard<std::mutex> lock(m_mutex);
            tmp = m_instance.load(std::memory_order_relaxed);
            if (tmp == nullptr) { // 第二次检查：防止多个线程排队通过了第一道关卡后重复 new
                tmp = new DclSingleton();
                m_instance.store(tmp, std::memory_order_release); // 释放栅栏，保证其他线程可见
            }
        }
        return tmp;
    }
};// 全局声明
std::atomic<DclSingleton*> DclSingleton::m_instance{nullptr};
std::mutex DclSingleton::m_mutex;
```
