## 📊 C++ 容器核心特性与复杂度全景表

| 容器 | 底层实现数据结构 | 是否支持随机访问 | 增 (插入) 复杂度 | 删 (删除) 复杂度 | 改/查 (访问) 复杂度 | 是否支持迭代器 (类型) | 核心常用场景 |
|---|---|---|---|---|---|---|---|
| vector | 连续动态数组 | 是 | 尾部 $O(1)$ amortized 中间/头部 $O(n)$ | 尾部 $O(1)$ 中间/头部 $O(n)$ | $O(1)$ | 是 (随机访问迭代器) | 默认首选；局部性好；频繁在尾部操作或随机读取的数据。 |
| deque | 分段连续数组 (中央控制图) | 是 | 头尾 $O(1)$ 中间 $O(n)$ | 头尾 $O(1)$ 中间 $O(n)$ | $O(1)$ | 是 (随机访问迭代器) | 需要在队列两端高效插入和删除（如实现双端队列、工作队列）。 |
| list | 双向链表 | 否 | 任意位置 $O(1)$ (已知迭代器) | 任意位置 $O(1)$ (已知迭代器) | $O(n)$ | 是 (双向迭代器) | 频繁在中间增删元素；需要拼接(splice)操作；不要求随机访问。 |
| forward_list | 单向链表 | 否 | 任意位置 $O(1)$ (已知迭代器后) | 任意位置 $O(1)$ (已知迭代器后) | $O(n)$ | 是 (前向迭代器，只可++) | 极致追求省内存的链表场景；元素量小且仅需单向遍历。 |
| array | 静态数组 (栈内存) | 是 | 不支持 | 不支持 | $O(1)$ | {是} (随机访问迭代器) | 代替传统 C 风格原生数组；大小固定；追求极致性能与安全性。 |
| set / map (含 multi) | 红黑树 (自平衡二叉搜索树) | 否 | $O(\log n)$ | $O(\log n)$ | $O(\log n)$ | 是 (双向迭代器) | 需要自动排序的去重集合(set)或键值对(map)；需要找邻近元素。 |
| unordered_set / map (含 multi) | 哈希表 (链地址法开链) | 否 | 平均 $O(1)$ 最坏 $O(n)$ | 平均 $O(1)$ 最坏 $O(n)$ | 平均 $O(1)$ 最坏 $O(n)$ | 是 (前向迭代器，只可++) | 高频面试必考！高效率的去重或查找，不需要关心元素顺序。 |
| stack | 容器适配器 (默认基于 deque) | 否 | 栈顶 $O(1)$ | 栈顶 $O(1)$ | 仅可访问栈顶 $O(1)$ | ❌ 否 | 严格后进先出 (LIFO)；DFS (深度优先搜索)；括号匹配；单调栈。 |
| queue | 容器适配器 (默认基于 deque) | 否 | 队尾 $O(1)$ | 队头 $O(1)$ | 仅可访问头尾 $O(1)$ | ❌ 否 | 严格先进先出 (FIFO)；BFS (广度优先搜索)；消息队列缓存。 |
| priority_queue | 二叉堆 (默认基于 vector) | 否 | $O(\log n)$ | $O(\log n)$ | 仅可访问堆顶 $O(1)$ | ❌ 否 | 动态获取最值；Top-K 问题；Dijkstra 算法优化。 |

## 💡 面试加分与避坑指南

   1. 关于迭代器失效 (Iterator Invalidation)：
   * vector：一旦扩容导致重新分配内存，所有迭代器全部失效；若未扩容，插入/删除点之后的迭代器失效。
      * list / set / map：插入操作从不导致迭代器失效；删除操作仅导致指向被删除元素的迭代器失效，其他不受影响。
   2. map 与 unordered_map 的选择题：
   * 不要盲目选 unordered_map。虽然其平均复杂度是 $O(1)$，但如果哈希冲突严重，会退化到 $O(n)$。此外，由于需要维护桶结构，它的内存占用明显高于红黑树的 map。
   3. 适配器屏蔽迭代器的原因：
   * 面试官常问：“为什么 stack 没有迭代器？” 答：为了保证接口的单一性和数据结构的绝对安全。如果开放迭代器，用户就能通过迭代器修改栈底或中间的元素，这违背了“后进先出”的封装原则。
