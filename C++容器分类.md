## 💡 针对 std::string 的大厂面试绝杀考点
将 std::string 纳入体系后，面试官最喜欢追问以下关于字符串底层内存的细节：

   1. 什么是 SSO（Short String Optimization）？
   * 回答：这是现代标准库对 std::string 做的一项重大内存优化。为了避免很短的字符串（如 "abc"、"hello"）也去高昂地申请堆内存，std::string 内部会自带一个大小通常为 15 或 16 字节的固定字符数组缓冲区（留在栈上）。当字符串长度未超过这个阈值时，直接存放在栈区；只有当字符串变长时，才会切换为传统的控制头结构，上堆申请连续空间。
   2. std::string 扩容时会发生什么？
   * 回答：与 vector 一致。当长度（size()）即将超过容量（capacity()）时，标准库会重新在堆上申请一块通常为 1.5 倍或 2 倍 的全新连续内存空间，将原本的数据拷贝或移动到新空间，然后把老内存释放掉。
   3. 有了 std::string 为什么 C++17 还要推 std::string_view？
   * 回答：因为 std::string 拥有内存。当它作为参数传递给函数时，如果对方传入的是 C 风格字面量 "hello"，就会隐式触发一次内存上堆、分配、拷贝的重资产行为。而 std::string_view 完全不拥有内存，它只是个指针加长度的轻量视图，实现了零开销传参和 $O(1)$ 复杂度的无拷贝切片。

## 💡 面试加分与避坑指南

   1. 关于迭代器失效 (Iterator Invalidation)：
   * vector：一旦扩容导致重新分配内存，所有迭代器全部失效；若未扩容，插入/删除点之后的迭代器失效。
      * list / set / map：插入操作从不导致迭代器失效；删除操作仅导致指向被删除元素的迭代器失效，其他不受影响。
   2. map 与 unordered_map 的选择题：
   * 不要盲目选 unordered_map。虽然其平均复杂度是 $O(1)$，但如果哈希冲突严重，会退化到 $O(n)$。此外，由于需要维护桶结构，它的内存占用明显高于红黑树的 map。
   3. 适配器屏蔽迭代器的原因：
   * 面试官常问：“为什么 stack 没有迭代器？” 答：为了保证接口的单一性和数据结构的绝对安全。如果开放迭代器，用户就能通过迭代器修改栈底或中间的元素，这违背了“后进先出”的封装原则。
