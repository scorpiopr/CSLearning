# C++ 智能指针用法
智能指针实现自动管理内存，包含在 <memory> 头文件中，分为三种类型：
## 1. std::unique_ptr (独占指针) —— 最常用
它是最轻量、效率最高的一种。正如其名，同一时间只能有一个 unique_ptr 指向该对象。

* 特性：
	* 不可拷贝（不能把一个 unique_ptr 赋值给另一个）。
   * 可移动（通过 std::move 将所有权转交给别人）。
   * 对象生命周期结束时，自动释放内存。
* 用法：

```
// 1. 创建（推荐使用 make_unique，C++14 起）
auto p1 = std::make_unique<int>(10);
// 2. 移动所有权（p1 变为空，p2 接管内存）
std::unique_ptr<int> p2 = std::move(p1);
// 3. 访问if (p2) 
cout << *p2;
// 4. 声明
unique_ptr<int> p1; // unique_ptr 默认初始化即为 nullptr，无需手动设置
```


## 2. std::shared_ptr (共享指针)
它允许多个指针同时指向同一个对象。

* 特性：
  * 引用计数：内部维护一个计数器，记录有多少个 shared_ptr 指向该对象。
   * 自动释放：当最后一个 shared_ptr 销毁（计数归零）时，才真正释放内存。
* 用法：

```
auto sp1 = std::make_shared<string>("Hello");
{
    auto sp2 = sp1; // 计数变为 2
    cout << sp1.use_count(); // 查看引用计数
} // sp2 离开作用域，计数变为 1// sp1 离开作用域后，内存释放
```


## 3. std::weak_ptr (弱指针)
它是 shared_ptr 的“助手”，不参与引用计数。

* 解决**循环引用**：如果两个类互相持有 shared_ptr，引用计数永远不会降到 0，导致内存泄漏。将其中一方改为 weak_ptr 即可解决。

    ```
    #include <iostream>
    #include <memory>
    class Boy; // 前置声明
    class Girl {
    public:
        std::shared_ptr<Boy> boyfriend; // 强引用指向 Boy
        ~Girl() { std::cout << "[Girl] 被安全析构了！\n"; }
    };
    class Boy {
    public:
        std::shared_ptr<Girl> girlfriend; // 强引用指向 Girl
        ~Boy() { std::cout << "[Boy] 被安全析构了！\n"; }
    };
    void memory_leak_leak_leak() {
        // 1. 在堆区创建两个对象，此时它们的强引用计数各自为 1
        auto tom = std::make_shared<Boy>();
        auto lucy = std::make_shared<Girl>();
    
        // 2. 🔑 致命互锁：让彼此内部的指针互相指向对方
        tom->girlfriend = lucy; // lucy 的强引用计数变为 2
        lucy->boyfriend = tom;  // tom 的强引用计数变为 2
        
        // 3. 函数结束，局部变量 tom 和 lucy 销毁（它们各自让对方的引用计数减 1）
        // 此时按理说应该触发析构，但奇迹（灾难）发生了……
    }
    int main() {
        memory_leak_leak_leak();
        std::cout << "函数结束了，堆内存真的释放了吗？\n";
        return 0;
    }
    ```

   1. 当函数结束时，栈上的局部变量自发析构，它们通过底层控制块将 Boy 和 Girl 的强引用计数各自从 2 扣减为 1。
   2. 智能指针触发释放的铁律是：“只有当强引用计数等于 0 时，才允许销毁对象并调用析构函数。”
   3. 此时，要想销毁 Boy 对象，就必须先销毁其内部的 girlfriend 指针；而要销毁 girlfriend 指针，就必须先释放 Girl 对象；而要释放 Girl 对象，又必须先销毁其内部的 boyfriend 指针……
   4. 逻辑陷入了经典的因果鸡蛋死循环。这就好比两个人互相抓着对方的衣领悬空挂在悬崖边，谁也不松手，结果就是一起在内存里永世僵死。

    改为使用std::weak_ptr 仅仅是一种对对象的“旁观监督者”，它只会指向对象，但绝对不会增加对象的【强引用计数】。它只能从一个 std::shared_ptr 或另一个 std::weak_ptr 中诞生。

  ```
  class Boy;
  class Girl {public:
      std::shared_ptr<Boy> boyfriend; // 依然是强引用
      ~Girl() { std::cout << "[Girl] 被安全析构了！\n"; }
  };
  class Boy {public:
      // 🔑 完美破局：改成弱引用！女朋友只是“旁观”男朋友，不强制捆绑他的生命
      std::weak_ptr<Girl> girlfriend; 
      ~Boy() { std::cout << "[Boy] 被安全析构了！\n"; }
  };
  void perfect_free() {
      auto tom = std::make_shared<Boy>();   // tom 强计数 = 1
      auto lucy = std::make_shared<Girl>(); // lucy 强计数 = 1
  
      tom->girlfriend = lucy; // 🔑 注意：因为是 weak_ptr，lucy 的强计数依然是 1！
      lucy->boyfriend = tom;  // tom 的强计数变为 2
  
      // 函数结束：
      // 1. 局部变量 lucy 销毁，让 Girl 的强计数归 0！Girl 当场引爆析构！
      // 2. Girl 析构的同时，会顺手将它持有的 boyfriend 指针销毁，让 Boy 的强计数从 2 跌落到 1。
      // 3. 局部变量 tom 销毁，让 Boy 的强计数从 1 归 0！Boy 当场引爆析构！
  }
  ```
* 安全观测：它能知道对象是否还存在，但不能直接操作。要使用时需通过 lock() 临时转回 shared_ptr。
  
  ```
  std::weak_ptr<string> wp = sp1;
  if (auto temp = wp.lock()) { // 如果对象还在，返回一个 shared_ptr
      cout << *temp;
  }
  ```

## 💡 核心规则与面试考点

1. 优先使用 unique_ptr：如果不需要多个指针共享同一个对象，首选 unique_ptr。因为它几乎没有性能开销。
2. 避免手动 new / delete：除非是在写底层数据结构（如刚才的 TrieNode 练习），否则在业务代码中应全面使用智能指针。
3. 传参建议：
 * 如果只是使用对象，不涉及所有权，函数参数直接传 原始指针 (ptr.get()) 或 引用 即可。
  * 不要轻易在函数参数里传 shared_ptr 的值，因为每次拷贝都会触发原子操作修改引用计数，影响性能。

## 总结比较

| 类型 | 拷贝性 | 所有权 | 性能 | 场景 |
|---|---|---|---|---|
| unique_ptr | ❌ 禁止 | 唯一拥有 | 极快（同原始指针） | 树节点、单例、局部资源管理 |
| shared_ptr | ✅ 允许 | 共同拥有 | 稍慢（有计数开销） | 缓存、多线程资源共享 |
| weak_ptr | ✅ 允许 | 不拥有 | 快 | 解决循环引用、缓存观察者 |
