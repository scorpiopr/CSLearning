# 重载运算符必须和用户定义的自定义类型的对象一起使用

------------------------------
# 一、 绝对【不能】重载的运算符（死记硬背考点）
C++ 中只有以下 5 个运算符被标准明确禁止重载：

   1. .（成员访问运算符）
   2. .*（成员指针访问运算符）
   3. ::（域解析/作用域运算符）
   4. ?:（三目条件运算符）
   5. sizeof（对象大小运算符）
   6. （现代 C++ 补充）：typeid（运行时类型识别运算符）和 noexcept（异常规格说明符）同样不可重载。

------------------------------
# 二、可以重载，但在实际工程开发中被严格禁止
在 C++ 中，逻辑运算符 &&（逻辑与）、||（逻辑或） 以及 ,（逗号运算符） 都是可以被重载的。但在绝大多数大厂的 C++ 编码规范（如 Google C++ Style Guide）中，都会有一条铁律：严禁重载 &&、|| 和逗号运算符。

## 1. 逻辑运算符 &&（逻辑与）、||（逻辑或）

------------------------------
### a. 致命缺陷：破坏了 C++ 的“短路求值（Short-Circuit Evaluation）”特性
在 C++ 原生语义中，逻辑运算符拥有非常聪明的“短路”保护机制：

* 对于 A && B：如果 A 为假，B 绝对不会被执行/计算。
* 对于 A || B：如果 A 为真，B 绝对不会被执行/计算。

这种机制常用于安全保护，例如：if (ptr != nullptr && ptr->value > 10)。如果 ptr 为空，右边的解引用就不会触发，程序不会崩溃。
### b. 重载后退化为普通函数调用
一旦你重载了 && 或 ||，它们就变成了普通的函数重载（形式上等价于 operator&&(A, B)）。
C++ 语法规定，在调用一个函数时，所有的参数必须先全部计算出来（求值），然后才能传进函数体内。

* 后果：无论左边的表达式是真是假，右边的表达式百分之百都会被强制执行。
* 灾难重现：

// 假设你为某个智能指针类重载了 operator&&if (my_ptr && my_ptr->isValid()) { 
    // ❌ 如果 my_ptr 是空的，由于重载失去了短路特性，
    // 右边的 my_ptr->isValid() 依然会被强制调用，直接引发段错误（Segfault）崩溃！
}


### c. 参数求值顺序变为了“未定义行为（Undefined Behavior）”
在 C++17 之前，函数参数的计算顺序是完全由编译器决定的（可能从左往右，也可能从右往左）。重载后，你不仅失去了短路特性，甚至连哪个表达式先执行都无法确定。

------------------------------
## 2、 逗号运算符（,）重载的灾难
逗号运算符的原本语义是：顺次执行多个表达式，并返回最后一个表达式的值。
如果重载了 ,它同样退化为函数调用，导致原本应该从左往右依次执行的表达式，求值顺序变得不可预测，这会直接摧毁像 for (int i = 0, j = 10; ...) 这样依靠逗号表达式确保执行顺序的常规代码逻辑。

------------------------------
# 三、 必须使用【类成员函数】重载的运算符
C++ 标准规定，以下 4 个运算符必须声明为类的非静态成员函数。如果试图将它们写成全局函数或友元函数，编译器会直接报语法错误（如 'operator...' must be a non-static member function）。

| 运算符 | 名称 | 必须写成成员函数的原因 |
|---|---|---|
| = | 赋值运算符 | 避免与编译器自动生成的默认赋值运算符产生语法二义性冲突。 |
| [] | 下标运算符 | 左操作数固定必须是该类对象本身，且需要支持左值修改逻辑。 |
| () | 函数调用运算符 | 用于定义仿函数（Functors），左操作数固定为对象实体。 |
| -> | 成员访问指针运算符 | 编译器对该运算符有特殊的连续解引用链式解析机制。 |

------------------------------
# 四、 在工程上必须使用【友元函数/全局函数】重载的运算符
从语法上讲，C++ 并没有强制规定哪些运算符必须用友元重载。但在实际工程落地与符合人类阅读习惯的考量下，以下两类运算符必须写成友元（全局）函数：
## 1. 流输入输出运算符（<< 和 >>）

* 核心原因：运算符的左操作数固定了其归属权。
* 原理解析：如果写成成员函数，左操作数必须是当前对象，调用会变成 obj << cout;，这严重违背了标准输入输出的直觉。为了让标准流对象站在左边（即 cout << obj;），该函数必须写在类外部（作为全局/友元函数），其第一个参数为`const std::ostream&`，第二个参数为自定义类对象，返回值类型为`std::ostream&`以保证可以连续输入输出（类似`cout<<a<<b`）。

## 2. 需要满足“对称性（隐式类型转换）”的双目算术运算符（如 +, -, *, /, ==）

* 原理解析：我们在复习手撕 Complex（复数类）时探讨过。如果把 + 写成成员函数，支持 c1 + 5.0，但绝对无法支持 5.0 + c1（因为左边是基本类型 double，无法调用成员函数）。为了让加号两边地位平等、能完美触发单参数构造函数的隐式转换，必须将其写成友元全局函数。

------------------------------

# 五、 核心规则对比表

| 运算符类别 | 典型代表 | 最佳重载方式 | 核心决定因素 |
|---|---|---|---|
| 赋值与行为类 | =, [], (), -> | 强制类成员函数 | C++ 标准强制约束，防止二义性 |
| 流交互类 | <<, >> | 必须友元/全局函数 | 左操作数是 ostream/istream 而非本类 |
| 对称算术类 | +, -, *, == | 推荐友元/全局函数 | 确保左右操作数地位平等，支持双向隐式转换 |
| 复合赋值类 | +=, -=, *=, /= | 推荐类成员函数 | 改变的是左操作数（对象自身）的状态，返回 *this |
| 自增自减类 | ++(前/后), -- | 推荐类成员函数 | 改变对象自身状态（***后置需加一个显式的 int 哑元参数占位***） |
