# 编译时多态 vs 运行时多态

| 对比维度 | 编译时多态（静态多态） | 运行时多态（动态多态） |
|---|---|---|
| 实现机制 | 函数重载、函数/类模板、CRTP | 虚函数（virtual）、继承、派生类重写（override） |
| 绑定时间 | 编译期（Compile-time） | 运行期（Runtime） |
| 底层原理 | 编译器名字修饰、模板实例化 | 虚函数表指针（vptr）和虚函数表（vtable） |
| 性能开销 | 无运行期开销。支持内联优化。 | 有虚函数表查找的时间开销，以及存储指针的空间开销。 |
| 核心局限 | 必须在编译时知道确定的类型。 | 可以在运行期动态处理未知类型的对象（只要它们继承自同一基类）。 |

# 编译时多态
在 C++ 中，编译时多态（Compile-time Polymorphism），也叫静态多态（Static Polymorphism）。
它是指在程序编译阶段，编译器根据函数调用时传入的参数类型、个数或者显式指定的模板参数，来决定具体调用哪一段代码。这种机制不需要在运行期通过指针查找，因此没有任何运行期开销，性能极高。
## 🛠️ 编译时多态的核心实现方式
在 C++ 中，编译时多态主要通过以下两种语法机制来实现：
## 1. 函数重载（Function Overloading）
在同一个作用域内，允许存在多个同名但参数列表不同（参数类型、个数或顺序不同）的函数。编译器在编译时会通过名字修饰（Name Mangling）机制，将这些同名函数转化为内部唯一的函数名，从而在编译阶段就绑定了具体的调用函数。
```
#include <iostream>
void print(int i) {
    std::cout << "Printing int: " << i << std::endl;
}
void print(double f) {
    std::cout << "Printing float: " << f << std::endl;
}
int main() {
    print(5);    // 编译时绑定到 print(int)
    print(5.5);  // 编译时绑定到 print(double)
}
```
## 2. 模板（Templates - 泛型编程）
通过函数模板或类模板，编写一段通用的代码骨架。当用户传入具体的类型时，编译器会在编译期进行模板实例化（Template Instantiation），为该类型自动生成一份专属的函数或类代码。

template <typename T>
T add(T a, T b) {
    return a + b;
}
int main() {
    add(1, 2);       // 编译时自动生成并调用 int add(int, int)
    add(1.5, 2.5);   // 编译时自动生成并调用 double add(double, double)
}

## 🚀 高级加分项：CRTP（奇异递归模板模式）
如果你在面试中提到了 CRTP (Curiously Recurring Template Pattern)，面试官会眼前一亮。CRTP 是一种利用类模板实现“不带虚函数表的高性能静态继承”的技术。它用编译时多态替代了传统的运行时多态：

template <typename Derived>class Base {public:
    void interface() {
        // 在编译期将 this 指针转换为派生类指针，调用派生类的方法
        static_cast<Derived*>(this)->implementation();
    }
};
class Derived : public Base<Derived> {public:
    void implementation() {
        std::cout << "Derived implementation" << std::endl;
    }
};


* 优点：实现了类似虚函数的接口继承效果，但由于是在编译期绑定，完全没有虚函数表的查找开销。

# 运行时多态实现机制
C++通过虚函数表实现动态绑定，根据不同对象调用不同的方法，具备代码复用，可扩展和类间解耦等优点。
## 一、 虛函數表和虚函数表指针
當一個類中包含了 virtual 關鍵字修飾的函數時，編譯器和運行期系統會在後台默默構建以下結構：
### 1.虚函数
* 在基类中声明一个函数为虚函数，使用关键字virtual。
* 派生类可以重写（override可省略，但不推荐）这个虚函数。
* 调用虚函数时，会根据对象的实际类型来决定调用哪个版本的函数。

### 2. 虛函數表 (vtable)

* 本質：一個存儲函數指針的數組。
* 分配在哪：分配在「文字常量區/只讀數據段（.rodata 段）」。
* 數量：每個包含了虛函數的「類」都有且僅有一張虛函數表。這張表是在編譯期就由編譯器完全確定並生成好的。

### 3. 虛函數表指針 (vptr)

* 本質：一個指向該類對象所對應的 vtable 的指針。
* 分配在哪：分配在「對象的內存空間中」，通常位於對象內存的最前端（Offset 0）。
* 數量：每個「對象實例」都包含一個 vptr。這個 vptr 是在對象構造函數執行期間被初始化賦值的。
* 大小：在 64 位系統下嚴格佔用 8 字节內存。

## 二、 動態綁定
动态绑定指程序运行期间确定调用哪个函数，需要使用***指向基类的指针或引用***来调用虚函数，编译器在运行时根据对象的实际类型来决定调用哪个函数
### 流程和内存分布

```
class Base {public:
    int m_val{1};
    virtual void func1() { std::cout << "Base::func1\n"; }
    virtual void func2() { std::cout << "Base::func2\n"; }
};
class Derived : public Base {public:
    int m_derived_val{2};
    void func1() override { std::cout << "Derived::func1\n"; } // 重寫了 func1
    // func2 未重寫，直接繼承 Base::func2
};
```

1. 當執行 Base* ptr = new Derived(); 時的內存分佈圖：

```
【 棧區 (Stack) 】         【 堆區 (Heap) —— Derived 對象實例 】
+---------------+        +-----------------------------------+

|  ptr (指針)   | ---->  | [Offset 0] vptr (虛表指針)        |
+---------------+        +-----------------------------------+

                         | [Offset 8] m_val = 1 (父類成員)   |
                         +-----------------------------------+

                         | [Offset 12] m_derived_val = 2     |
                         +-----------------------------------+

                                         |
                                         v 指向
                        【 只讀數據區 (.rodata) —— Derived 的 vtable 】
                        +------------------------------------+
                        | Slot 0: &Derived::func1 (重寫後的)  |
                        +------------------------------------+

                        | Slot 1: &Base::func2    (繼承父類的) |
                        +------------------------------------+
```

2. 動態綁定的三步跳躍（ptr->func1() 的運行期原理）
當代碼執行 ptr->func1(); 時，編譯器不會像普通函數那樣直接生成調用某個固定地址的 call 指令，而是將其轉化為以下底層邏輯：

   1. 第一步：訪問 ptr 指向的對象，取出該對象頭部的 vptr 的地址。
   2. 第二步：順著 vptr 找到 Derived 類的虛函數表（vtable）。
   3. 第三步：根據編譯期就定好的索引（func1 佔用 Slot 0），取出該位置上的函數指針 &Derived::func1，最後跳轉過去執行。这个過程就叫運行期動態綁定。

### 类指针和对象
虚函数表指针在对象创建时确定，不随指向对象的指针的类型变化。
```
#include <iostream>
using namespace std;
class A
{   
    public: void virtual f() 
    {      
        cout << "A" << " ";
    }
};
class B : public A
{   
    public: void virtual f()
    {      
        cout << "B" << " ";
    }
};
int main(){
    A *pa = new A();
    pa->f();
    B *pb=(B *)pa;
    pb->f();    
    delete pa, pb;
    pa=new B();
    pa->f();
    pb=(B *)pa;
    pb->f();
    return 0;
}
```

## 四、注意
## 1：對象切片（Slicing）為什麼會導致多態失效？

* 回答：當你寫 Base b = d;（用基類對象接收派生類對象）時，b 是一個分配在棧區的純父類對象。編譯器只會拷貝 d 對象中屬於父類的成員變量，並且會把 b 內存頭部的 vptr 強制初始化並指向 Base 類的虛函數表。既然 vptr 指向了父類表，查表出來的永遠是父類的函數，多態自然失效。

## 2：構造函數體內調用虛函數，會觸發多態嗎？

* 回答：不會。在執行基類的構造函數時，子類特有的成員還沒有開始初始化。為了防止子類虛函數意外讀取到子類的未初始化垃圾數據，C++ 語法規定，在基類構造函數執行期間，對象的 vptr 暫時只指向基類的 vtable。此時調用虛函數，只會觸發靜態綁定，調用父類自己的版本。

## 3：多重繼承下，一個對象有幾張虛函數表、幾個 vptr？

* 回答：如果類 C 同時繼承自 A 和 B（且 A、B 都有虛函數），那麼對象 c 內部會包含 2 個獨立的 vptr（一個對齊 A 的偏移，一個對齊 B 的偏移），對應的類 C 在編譯期也會生成 2 張不同的虛函數表，編譯器會在調用時自動進行指針偏移計算（Thunk 技術）來確保正確調用。
## 4：私有成员函数可以命名为虚函数
* 如果派生类重写了私有虚函数,通过基类指针调用时仍然会表现出多态性。

## 5：override可否省略
在 C++ 中，派生类重写基类的虚函数时，可以省略 override，但可能造成隐式重载（Overload）。
例如：
```
class Base {public:
    virtual void process(int x) const { /* ... */ }
};
class Derived : public Base {public:
    // ❌ 灾难现场：开发者不小心漏掉了 const 关键字，且省略了 override
    void process(int x) { 
        // 你的本意是想重写父类的逻辑
    } 
};
```

* 致命真相：由于漏掉了 const，子类函数的签名和父类其实不一致。
* 编译器的误判：此时，编译器翻遍虚表，认为你根本不是在重写父类，而是想在子类里自娱自乐地创造一个全新的、普通的成员函数（构成函数重载）。因为你没有写 override，编译器觉得这一切非常合理，完全不报任何警告，直接通过编译！
* 运行期雪崩：当你用基类指针调用 Base* ptr = new Derived(); ptr->process(10); 时，程序在运行期通过虚表查到的依然是父类的旧逻辑，子类辛辛苦苦写的核心业务代码一次都不会被执行！这种 Bug 在大型项目中极难排查。

强制加上 override 关键字，上面的悲剧会如何被完美化解？

class Derived : public Base {public:
    // ✅ 现代标准写法：强行加上注解
    void process(int x) override { 
        // ...
    } 
};

* 编译期的绝对防御：一旦你加上了 override，等于对编译器下达了死命令：“帮我死死盯着，这个函数在父类里必须有一个一模一样的虚函数存在！”
* 当编译器发现子类由于漏掉 const 或写错了一个参数类型导致签名对不上时，会当场拒绝编译并抛出致命错误：error: 'void Derived::process(int)' marked 'override' does not override。
