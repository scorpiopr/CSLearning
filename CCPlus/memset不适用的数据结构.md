所有包含非平凡（Non-trivial）底层结构、动态内存管理或非 0/ -1 初始化需求的复杂数据结构，都不适用 memset。
如果强行使用 memset，轻则导致数据损坏、内存泄漏，重则引发程序崩溃（Segmentation Fault）。
以下是具体不适用的数据结构和场景分类：
------------------------------
## 1. 包含虚函数的类对象（Virtual Functions）

* 代表结构：自定义的 C++ 类，且类中包含 virtual 关键字（如虚函数、虚析构函数）。
* 原因：这类对象在内存中的头部（或尾部）隐式包含一个虚函数表指针（vptr） [fanyv88.com]。memset 会把这个 vptr 强行清零。
* 后果：一旦 vptr 变成 0x0，后续只要发生多态调用（如 obj->f()），程序就会因为找不到虚函数表而直接崩溃。

## 2. 任何标准的 C++ STL 容器

* 代表结构：std::string, std::vector, std::list, std::map, std::unordered_map 等。
* 原因：STL 容器内部通常包含指向动态分配内存的指针、大小计数器（size/capacity）或者复杂的树状/哈希内部结构。
* 后果：
* 内存泄漏：强行 memset 会直接抹除指向实际数据内存的指针，导致原先分配的内存在底层无法被释放。
   * 结构破坏：破坏了容器的内部元数据，一旦容器在后续执行析构、扩容（push_back）或查询时，会直接引发未定义行为（Undefined Behavior）。

## 3. 包含指针或引用成员的结构体

* 代表结构：
```cpp
struct Node {
    int id;
    char* name; // 包含指针
};
```
* 原因：memset 只能作用于 Node 本身占用的内存（在栈上或堆上固定大小的空间），它不会顺着 name 指针去清理它指向的字符串字符串。
* 后果：将 name 指针强行变成了 NULL，导致原本通过 malloc 或 new 分配给名字的内存空间彻底丢失并发生泄漏。

## 4. 浮点数数组（当初始化非 0 时）

* 代表结构：float arr[100]; 或 double arr[100];
* 原因：浮点数在计算机中遵循 IEEE 754 标准，其内存结构分为符号位、指数位和尾数位。
* 后果：如果你尝试用 memset(arr, 1, sizeof(arr)) 把浮点数数组初始化为 1.0，每一个字节都会被填入 0x01。这在 IEEE 754 映射下会变成一个极其微小的、接近于 0 的非正规浮点数，而绝不是 1.0。
(注：memset 给浮点数初始化为 0 或 -0.0 是可行的)

## 5. 整型数组初始化为非 0 和 -1 的值

* 代表结构：int arr[100];
* 原因：memset 是逐字节填充。一个 int 占用 4 个字节。
* 后果：如果你执行 memset(arr, 1, sizeof(arr))，你期望每个元素是 1，但实际上每个 int 的 4 个字节都被填成了 0x01，即二进制的 00000001 00000001 00000001 00000001。这转换成十进制是 16843009，而不是 1。
(注：只有 0（0x00000000）和 -1（补码 0xFFFFFFFF）可以通过逐字节填充正确实现)

------------------------------
## 🛡️ 现代 C++ 的安全替代方案
在现代 C++（C++11 及更新版本）中，应该尽量避免使用传统的 memset，推荐以下更安全、具备类型安全检查的方法：

* 零初始化（推荐）：在定义时直接使用 {}。
```cpp
MyStruct obj{}; // 所有成员自动执行零初始化/默认构造int arr[100]{}; // 数组全清零

* 填充任意值：使用 <algorithm> 库中的 std::fill 或 std::fill_n。

#include <algorithm>int arr[100];std::fill(std::begin(arr), std::end(arr), 42); // 每一个元素都安全地变成 42
```
