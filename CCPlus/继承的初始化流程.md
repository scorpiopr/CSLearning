## 💻 经典代码实战验证
你可以通过以下标准 C++ 代码，直观地在终端中观察到这 8 个步骤的精确打印顺序：
```cpp
#include <iostream>
class BaseComponent {
public:
    BaseComponent() { std::cout << "[1] 基类的成员对象 构造\n"; }
    ~BaseComponent() { std::cout << "[8] 基类的成员对象 析构\n"; }
};
class DerivedComponent {
public:
    DerivedComponent() { std::cout << "[3] 派生类的成员对象 构造\n"; }
    ~DerivedComponent() { std::cout << "[6] 派生类的成员对象 析构\n"; }
};
class Base {private:
    BaseComponent bc; // 基类的成员对象public:
    Base() { std::cout << "[2] 基类 自身的构造函数体\n"; }
    virtual ~Base() { std::cout << "[7] 基类 自身的析构函数体\n"; }
};
class Derived : public Base {
private:
    DerivedComponent dc; // 派生类的成员对象public:
    Derived() { std::cout << "[4] 派生类 自身的构造函数体\n"; }
    ~Derived() override { std::cout << "[5] 派生类 自身的析构函数体\n"; }
};
int main() {
    std::cout << "--- 🔍 构造阶段开始 ---\n";
    Base* ptr = new Derived();

    std::cout << "\n--- 🔍 析构阶段开始 ---\n";
    delete ptr;
    
    return 0;
}
```
## 📊 终端运行输出结果

--- 🔍 构造阶段开始 ---
[1] 基类的成员对象 构造
[2] 基类 自身的构造函数体
[3] 派生类的成员对象 构造
[4] 派生类 自身的构造函数体

--- 🔍 析构阶段开始 ---
[5] 派生类 自身的析构函数体
[6] 派生类的成员对象 析构
[7] 基类 自身的析构函数体
[8] 基类的成员对象 析构

### 1. 构造流程（全面拆解：共 4 大步）
当实例化派生类对象时，底层的构造顺序严格如下：
```
[开始构造 Derived 对象]
   │
   ├── 步骤 1：进入基类（Base）的初始化阶段
   │     └── 1.1 构造【基类的成员对象】 (BaseComponent)
   │
   ├── 步骤 2：执行【基类】自身的构造函数体 (Base::Base)
   │     └── *(此时对象的 vptr 指向 Base 虚函数表)*
   │
   ├── 步骤 3：进入派生类（Derived）的初始化阶段
   │     └── 3.1 构造【派生类的成员对象】 (DerivedComponent)
   │
   └── 步骤 4：执行【派生类】自身的构造函数体 (Derived::Derived)
         └── *(此时对象的 vptr 彻底更新，指向 Derived 虚函数表)*
[Derived 对象构造完毕]
```
### 2. 析构流程（全面反转：共 4 大步）
当销毁该派生类对象时，析构顺序与构造完美相反：
```
[开始销毁 Derived 对象]
   │
   ├── 步骤 1：执行【派生类】自身的析构函数体 (~Derived)
   │     └── *(此时开始，子类虚函数机制失效)*
   │
   ├── 步骤 2：销毁【派生类的成员对象】 (~DerivedComponent)
   │
   ├── 步骤 3：执行【基类】自身的析构函数体 (~Base)
   │     └── *(基类的析构函数必须是 virtual)*
   │
   └── 步骤 4：销毁【基类的成员对象】 (~BaseComponent)
[对象内存完全释放]
```

## ⚠️ 面试官必挖的 3 个隐藏细节

   1. 多个成员对象的构造顺序
   如果 Derived 类里面同时包含 DerivedComponent1 dc1; 和 DerivedComponent2 dc2;，它们的构造顺序由它们在类内部声明的先后顺序决定，而不是你在初始化列表（Initializer List）里写的顺序。
   2. 初始化列表的本质
   在步骤 1.1 和 3.1 中，成员对象的构造是在当前类的构造函数体执行之前完成的。如果你没有在初始化列表中显式调用成员对象的带参构造函数，编译器会自动调用它们的默认构造函数。
   3. 安全防线：析构顺序的必然性
   为什么派生类的成员 dc 要比基类 Base 先析构？因为派生类的成员对象可能在生命周期中依赖了基类的资源（例如基类里有个数据库连接指针或日志句柄）。如果先析构基类，那么派生类成员在析构时一旦尝试访问这些已被销毁的基类资源，就会引发野指针崩溃。
