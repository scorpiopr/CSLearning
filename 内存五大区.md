# 一、 内存五大区核心对比表

| 比较维度 | 栈区 (Stack) | 堆区 (Heap) | 全局/静态区 (Global/Static) | 文字常量区 (Constant) | 代码区 (Code/Text) |
|---|---|---|---|---|---|
| 主要存储内容 | 局部变量、函数参数、返回地址、寄存器上下文 | new/malloc 动态开辟的数据、大对象、动态数组 | 全局变量、静态变量（static） | 字符串字面量、编译期确定的常数 | 函数体的二进制机器指令、成员函数代码 |
| 生命周期 | 伴随函数或局部作用域（进入时创建，离开时销毁） | 程序员手动控制（从 malloc/new 到 free/delete） | 伴随整个程序始终（程序启动到进程结束） | 伴随整个程序始终 | 伴随整个程序始终 |
| 作用域 | 局部作用域（仅在定义的函数或 {} 代码块内有效） | 整个程序（通过指针在整个程序内传递和访问） | 根据修饰符决定： • 普通全局：全程序 • 静态全局：当前源文件 • 局部静态：局部函数内 | 整个程序（只读访问） | 整个程序（只读执行） |
| 初始化 | 不自动初始化（若不赋值则为随机垃圾值） | 不自动初始化（calloc 除外，若不手动赋值则为垃圾值） | 自动初始化： • 已初始化存入 .data • 未初始化存入 .bss（自动清零） | 编译时完成硬编码固化 | 编译时完成硬编码固化 |
| 分配/释放方式 | 编译器自动分配与释放（出栈入栈） | 程序员手动申请和释放（C++11 后可由智能指针托管） | 编译器在程序启动时分配，系统在退出时释放 | 编译器分配，系统在进程结束时释放 | 编译器分配，系统在进程结束时释放 |
| 分配速度 | 极快（CPU 设有专门的栈指针寄存器，单条指令操作） | 较慢（涉及内核调用，需在内存碎片中查找合适空闲块） | 极快（在程序加载时一次性分配完毕） | 极快（程序加载时一次性调入内存） | 极快（程序加载时一次性调入内存） |
| 空间大小 | 非常有限（Linux 默认 8MB，Windows 默认 1~2MB） | 非常大（受限于操作系统的虚拟内存和物理内存总量） | 固定大小（取决于代码中全局和静态变量的总量） | 固定大小（取决于代码中常量的总量） | 固定大小（取决于生成的二进制机器代码大小） |
| 内存连续性 | 绝对连续（由于栈的先进后出特性，内存紧密排列） | 不连续（频繁申请释放会产生大量内存碎片，呈链表状） | 连续（各个变量在 .data 或 .bss 段中紧密排列） | 连续（在 .rodata 段中紧密排列） | 连续（在 .text 段中紧密排列） |
| 内存增长方向 | 向低地址增长（高地址 $\rightarrow$ 低地址，向下） | 向高地址增长（低地址 $\rightarrow$ 高地址，向上） | 固定大小，不动态增长 | 固定大小，不动态增长 | 固定大小，不动态增长 |
| 读写权限 | 可读可写 | 可读可写 | 可读可写 | 只读（强制修改会引发段错误 Segmentation Fault） | 只读（具有内核保护，防止代码被运行时篡改） |
| 常见引发的 Bug | 递归过深导致栈溢出（Stack Overflow） | 忘记释放导致内存泄漏；未判空导致空指针异常 | 跨文件动态初始化顺序不确定导致死锁或读到垃圾值 | 强行通过指针修改字面量导致程序崩溃 | 极少出错（若损坏通常是硬件或黑客缓冲区溢出攻击） |

# 二、全局变量与局部静态变量的初始化时机
| 变量类型 | 初始化时机 | 初始化发生的阶段 | 多线程安全性 |
|---|---|---|---|
| 全局变量 (Global) | 程序主函数 main() 执行之前 | 编译期（常量）或 运行时启动阶段（动态） | 天然安全（此时只有一个主线程，还没开始多线程） |
| 局部静态变量 (Function Static) | 该函数第一次被调用、且代码运行到该变量声明处时 | 运行时（Runtime）首次调用阶段 | 现代 C++ 安全（C++11 保证原子性） 传统 C/C++ 不安全 |


## 1、 全局变量的初始化时机
全局变量在 main() 函数执行前就已经完成了初始化。根据初值的内容，编译器和操作系统将其分为两步处理：

   1. 静态初始化（Static Initialization）：
   * 如果全局变量使用的是编译期常量（如 int g_val = 100;）或者没有赋初值（默认初始化为 0），它在编译时就被直接硬编码进了二进制文件的 .data 段（已初始化数据区）或 .bss 段（未初始化数据区）。程序被操作系统加载进内存时，这些值就已经存在了。
   2. 动态初始化（Dynamic Initialization）：
   * 如果全局变量需要调用函数或构造函数来初始化（如 int g_val = get_initial_value();），编译器会在 main() 执行前，通过一段隐式的启动代码（由标准库 crt0 提供）去依次调用这些函数或构造函数。
   
⚠️ 大厂常考深坑：跨文件全局变量初始化顺序不确定（Static Initialization Order Fiasco）
如果 fileA.cpp 里的全局变量 g_A 的初始化依赖 fileB.cpp 里的全局变量 g_B，由于 C++ 标准没有规定不同源文件之间全局变量的动态初始化顺序，这极有可能导致 g_A 初始化时 g_B 还是个垃圾值。
工程解决方案：利用下面要讲的“局部静态变量”来实现迈耶斯单例模式（Meyers Singleton）。

## 2、局部静态变量的初始化时机
局部静态变量（在函数内部声明的 static 变量）具有局部作用域，但拥有全局生命周期。

```
void test_func() {
    // 初始化时机：第一次调用 test_func() 运行到这一行时
    static int local_static_val = get_value(); 
    std::cout << local_static_val << std::endl;
}
```


   1. 首次调用初始化：
   当程序第一次调用 test_func() 且执行到这一行时，操作系统和运行时环境才会去调用 get_value() 并将其赋值给 local_static_val。
   2. 后续调用直接跳过：
   当第二次、第三次调用该函数时，初始化代码会被直接跳过，变量会保持上一次调用结束后的值。
   3. 底层如何知道是“第一次”？：
   编译器在底层为每个局部静态变量暗中维护了一个隐藏的 布尔标志位（Guard Variable）。伪代码逻辑如下：
   
```
   // 编译器的底层视角
	 if (!__guard_flag_is_initialized) {
       local_static_val = get_value();
       __guard_flag_is_initialized = true;
   }
```

## 3、局部静态变量的线程安全
在 C++11 之前，上述伪代码中的 if 判断在多线程并发环境下是不安全的。如果两个线程同时第一次调用该函数，可能会导致该变量被初始化两次，或者一个线程拿到了初始化到一半的脏数据。
从 C++11 开始（当然包括你需要的 C++17），标准做出了解释和强制规定：局部静态变量的初始化过程必须是线程安全的（Thread-safe Base Initialization）。

* 底层实现：编译器会在操作那个隐藏的标志位时，自动加上互斥锁（Mutex）或原子操作。
* 完美应用：这一特性让 C++ 实现单例模式变得极其优雅和安全（被称为 Meyers Singleton）：

```
class Singleton {public:
    static Singleton& getInstance() {
        // C++11/17 保证：多线程并发时，下行代码的初始化绝对是原子性、线程安全的
        static Singleton instance; 
        return instance;
    }private:
    Singleton() = default; // 构造函数私有化
};
```


# 三、 典型示例
通过一段经典的 C++ 代码，来直观看看各个变量究竟被安置在哪个区：

```
#include <iostream>
#include <vector>
int g_var1 = 10;          // 1. 全局/静态区（.data 段：已初始化）
int g_var2;               // 1. 全局/静态区（.bss 段：未初始化，自动清零）
static int s_g_var = 20;  // 1. 全局/静态区（内部链接静态变量）
void demoFunction(int param) { // 2. param 在栈区
    int local_var = 5;          // 2. local_var 在栈区
    static int s_local_var = 1; // 1. 全局/静态区（函数局部静态变量）
    
    // 3. str_ptr 变量本身在栈上
    // 4. 但它指向的 "Hello World" 字符串存放在【文字常量区】
    const char* str_ptr = "Hello World"; 
    
    // 5. ptr 变量本身在栈上
    // 6. 但 `new` 开辟的 40 字节内存空间在【堆区】
    int* ptr = new int[10]; 
    
    delete[] ptr; // 释放堆内存
} // 函数结束：param、local_var、str_ptr、ptr 本身被栈弹出销毁
```

# 四、 面试高频核心追问
## 问 1：栈和堆的增长方向为什么是相对的？
答：这是一种精妙的内存利用设计。在进程的虚拟内存空间中，栈位于高地址向下增长，堆位于低地址向上增长。让它们“迎面相向”发展，可以最大限度地利用中间的自由存储空间。无论是栈临时需要极多空间，还是堆需要大量开辟，只要两者不相遇（即不发生碰撞），内存就能被高效动态地共享。
## 问 2：执行 int* p = new int(5); 这行代码时，内存是如何布局的？
答：

   1. 指针变量 p 作为一个局部变量，其本身（占用 8 字节，在 64 位系统下）被分配在栈区。
   2. 编译器调用 operator new，在堆区成功申请了一块存放单个整数（4 字节）的内存，并把数值 5 写入该堆内存。
   3. 最后，将堆区这块内存的首地址赋值给栈区的指针 p。
 ## 问3：.data 与 .bss 的区别
 全局/静态区在最终的二进制文件中被分成两个段。.bss 段（Block Started by Symbol）只记录变量名和大小，不占用实际的磁盘二进制文件空间，只有当程序被加载到内存时，操作系统才会给 .bss 分配空间并全部清零，这极大地减小了编译出来的可执行文件体积。
